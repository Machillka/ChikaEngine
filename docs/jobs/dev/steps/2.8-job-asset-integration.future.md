# Step 2.8 - Asset Async Integration

## Metadata

- Status: Complete (2026-06-20)
- Depends on: Step 2.7
- Estimate: 1-2 days

## Goal

用 Job System 替换 AssetManager 零散 `std::async`，证明调度器能够服务真实引擎工作负载。

## Responsibility

- Asset Job 只执行磁盘读取、解析和 CPU 数据构建。
- AssetManager 保持缓存和 Handle 所有权。
- ResourceManager/GPU Upload 留在 Render/Main 线程边界。

## Path Plan

- `AssetManager.hpp/.cpp`
- `RenderResourceRequest.hpp`
- `EngineContext.cpp`
- `tests/integration/AssetJobIntegrationTests.cpp`

## Implementation

- AssetManager 注入 `JobSystem*`，不访问全局单例。
- `Load*Async` 返回引擎 Future/AssetRequest 或带状态 Handle。
- 相同 GUID 并发请求去重。
- Shutdown Drain/Cancel 与 AssetManager 生命周期一致。
- 完成 CPU Load 后发布主线程/GPU upload request。

## Unit Tests

- 并发请求同一资产只加载一次。
- 多资产并行完成且 Handle 有效。
- Loader 异常/缺失文件传播失败。
- Shutdown 期间请求不访问已销毁 AssetManager。
- GPU API 不在 Worker thread 调用。

## Gate

Job System 通过正确性测试并成功驱动 Asset 工作负载，才能进入 Renderer 并行化。

## Implementation Record

- `AssetManagerCreateInfo` 显式注入 JobSystem；原有 `shared_future<Handle>` API 保持兼容，内部不再创建 `std::async` 线程。
- 同路径成功请求复用 Future/CPU Handle；失败请求移除后可重试，Unload 清理对应 Future，Loader 异常写入 Future。
- CPU loader 在 Worker 执行；SlotMap/cache 提交受 Asset mutex 保护，ResourceManager/RHI 上传仍留在主/渲染线程。
- Renderer 仅把主视图与阴影视图两个只读 Visibility 任务接入 Jobs；资源查询、Queue 构建和 RHI 不进入 Worker，调度失败回退串行。

## Changed Files and Reasons

- `AssetManager.hpp/.cpp`、`ShaderLoader.cpp`：统一异步调度、请求去重、错误传播和缺失 Shader 失败语义。
- `EngineContext.cpp`：按 Scene/Renderer/Asset/Jobs 依赖逆序关闭。
- `RenderPipeline.hpp/.cpp`：证明不可变 RenderWorldSnapshot 可被并发读取，但不提前实现 Phase 3 分块 Renderer。
- `tests/integration/AssetJobIntegrationTests.cpp`、`tests/unit/RenderBaselineTests.cpp`：覆盖真实资产与双 View 可见性结果一致。

## Knowledge and Design Notes

- Job System 管“何时在哪个 CPU Worker 执行”，AssetManager 仍管缓存、Handle 和请求去重；调度器不能接管资产所有权。
- GPU API 的线程亲和性没有因 CPU Job 引入而改变，Worker 只能产出 CPU 数据或主线程请求。
- 当前无 JobSystem 的 AssetManager 使用同步 fallback，目的是兼容独立测试且不再偷偷创建线程；真正 IO 专用池仍是后续项。
