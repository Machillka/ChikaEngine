# Step 2.2 - Job Storage and Generation Handle

## Metadata

- Status: Complete (2026-06-20)

## Goal

使用 Pool 保存 Job 状态，并以 index + generation Handle 替代裸指针和无界 Future。

## Responsibility

- Job Pool 拥有 callable、状态、异常和 completion counter。
- JobHandle 只标识，不拥有 Job。
- 回收只发生在完成且无内部引用后。

## Path Plan

- `JobStorage.hpp/.cpp`
- `JobState.hpp`
- `SmallJobFunction.hpp`

## Implementation

- Handle index/generation/invalid。
- Job 状态 Free、Created、Queued、Running、Completed、Cancelled、Failed。
- Small callable 内联存储；超限行为明确。
- Exception 保存为 `exception_ptr`，由 Wait/Rethrow 查询。

## Unit Tests

- 创建/完成/回收后旧 Handle 失效。
- generation wrap 策略有测试或静态边界。
- Callable 正常析构且不重复执行。
- 抛异常 Job 进入 Failed 并可传播。
- Pool 容量耗尽返回明确错误。

## Implementation Record

- `JobStorage` 预分配固定 Slot；`JobHandle` 使用 32-bit index + 32-bit generation，回收后旧 Handle 失效。
- 常规 callable 使用 128-byte move-only `SmallJobFunction` 内联保存；完成、取消和异常均为显式终态。
- Handle 由调用方 `Release`，fire-and-forget Job 用 `Detach` 把回收责任交还调度器。

## Changed Files and Reasons

- `JobTypes.hpp`、`JobDesc.hpp`：定义 Handle、状态、目标和失败策略。
- `JobStorage.hpp/.cpp`、`SmallJobFunction.hpp`：实现有界存储、generation 校验和无常规 Job 级堆分配。

## Knowledge and Design Notes

- Handle 不是所有权对象；generation 解决 Slot 重用后的 ABA 身份混淆，但不能跨进程持久化。
- 固定容量把内存上限和容量失败变成可测试契约；超大捕获应改为共享请求状态，而不是扩大所有 Job Slot。

## Verification Record
