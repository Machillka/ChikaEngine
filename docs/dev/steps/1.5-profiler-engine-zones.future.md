# Step 1.5 - Engine CPU Zone Integration

## Metadata

- Status: Complete (2026-06-19)

## Goal

在不改变行为的前提下，为主循环和关键 Subsystem 增加第一批稳定 Zone。

## Responsibility

- 各模块只声明自身高层 Zone 和 Counter。
- Profiler 不调用模块内部逻辑。
- Zone 名称作为分析 API，避免每次重命名。

## Implementation

- Frame、Application Update、Scene Tick、Fixed Tick。
- Physics、Animation、Render Bridge。
- Snapshot、Visibility、Packet Build、Sort/Batch、RenderGraph Compile/Execute。
- Counter：objects、visible、packets、batches、draws。

## Unit Tests

- 一个 Headless/Mock 帧包含预期顶层 Zone。
- Zone 顺序与生命周期一致。
- Profiling Off 时现有测试行为不变。
- 不允许单对象循环产生无界 Zone 数。

## Implementation Record

- 主循环以 frame ID 建立边界，覆盖 Application update、Engine tick、Scene/FixedUpdate、Physics、Animation、RenderWorld snapshot 与 Renderer 生命周期。
- RenderPipeline 覆盖 frame data、resource、visibility/queue、instance、graph build/execute，并发布 pass/draw/instance/visible/culled/batch counters。
- 保持阶段级粒度，不在 GameObject 或 Draw 循环内为每个对象生成动态 zone。

## Changed Files and Reasons

- `Application.cpp`、`EngineContext.cpp`：定义跨模块顶层时序。
- `Framework` Scene/Subsystem：标记 Gameplay 与 RenderWorld bridge 成本。
- `Render` Renderer/Pipeline/Visibility/Queue/Graph：建立 Renderer CPU 分解。

## Knowledge and Design Notes

- Zone 名称是性能分析接口；稳定命名比直接使用函数名更适合跨版本对比。
- Counter 必须在 RenderFrameStatistics 汇总后采样，否则 UI 中的 zone 与统计会来自不同帧阶段。

## Verification Record

- `ChikaRenderBaselineTests` 用 Mock RHI 验证真实 `RenderGraph.Compile/Execute` zone 进入 frame capture。
- Debug `ChikaEngine`、`ChikaEditor` 和 Release profiler benchmark 均构建通过。
