## Goal

以固定种子程序化生成 Instance、State Diversity 和 Dynamic Transform 三类场景。

## Responsibility

- Scene Factory 只创建确定性 Render/Game 数据。
- Benchmark Runner 决定对象数量和运行时长。
- 资产仍通过正式 AssetReference/AssetManager 加载。

## Implementation

- `instance-{1k,10k,50k,100k}`：相同 Mesh/Material、规则网格布局。
- `state-diversity`：固定比例 Mesh/Material/Pipeline Key。
- `dynamic-5k`：固定轨迹更新 Transform。
- 固定 Camera、随机种子、对象 ID 顺序和场景配置 Hash。

## Unit Tests

- 同 seed/config 生成相同对象数、Transform Hash 和资产引用。
- 不同对象档位数量准确。
- Dynamic 场景在同一 frame index 产生相同 Transform Hash。
- 未找到基准资产时快速失败。

## Implementation Record

- `BenchmarkSceneLayout` 纯 CPU 生成 1K/10K/50K/100K、state-diversity 与 dynamic-5k，固定 seed、网格、Camera 规则和 FNV-1a 配置 Hash。
- `BenchmarkSceneFactory` 只使用 AssetDatabase 中的正式 GUID 引用，先验证全部 fixture，再创建 Scene Camera、Light 和 MeshRenderer。
- Dynamic 场景按绝对 simulation frame 计算轨迹，不累积 `deltaTime` 误差。
- 场景构建默认 120 秒超时，每 256 个对象检查 deadline；100K 已完成短 smoke。
- 高吞吐测试发现并修复 Snowflake UID 的 machine/sequence 位重叠，50K 场景现可稳定保持 50K 唯一对象。

## Changed Files and Reasons

- `BenchmarkSceneLayout.hpp/.cpp`：把确定性负载描述与 Framework 对象创建分离，便于无窗口单测。
- `BenchmarkSceneFactory.hpp/.cpp`：负责正式资产引用、Scene 实体化、动态对象索引和超时失败。
- `UIDGenerator.h`：`machineShift` 改为 `sequenceIDBits`，避免 12-bit sequence 与 machine id 重叠。
- `tests/unit/BenchmarkTests.cpp`、`CoreBoundaryTests.cpp`：验证对象数、Hash、动态轨迹、缺失资产和 50K burst ID 唯一性。

## Knowledge and Design Notes

- **确定性不等于固定随机种子**：Camera、资产身份、对象顺序、时间推进和运行模式也必须固定，Hash 用于证明这些输入一致。
- **布局与实体化分层**：布局是可测试的数据生成器；Factory 才接触 AssetManager/Scene，类似把 workload description 与 runtime adapter 分开。
- **GUID 而非路径身份**：路径只用于发现 fixture，创建组件时保存 GUID，避免移动资产后 benchmark 语义改变。
- **压力测试也是正确性测试**：50K 场景揭露了低规模单测无法发现的 UID 位域碰撞，说明 benchmark 不只用于“跑分”。
