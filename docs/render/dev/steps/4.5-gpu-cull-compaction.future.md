# Step 4.5 - GPU Frustum Culling and Visible Compaction

## Metadata

- Status: Complete (CPU Oracle + Shader Contract, 2026-06-26)
- Depends on: Step 4.4
- Estimate: 2-3 days

## Goal

在 Compute Pass 中完成 Frustum Culling，并把可见 Instance 压缩为按 Draw Group 可消费的列表。

## Responsibility

- Compute Shader 负责 bounds test、counter 和 compaction。
- CPU 负责上传 frustum、instance count 和 group metadata。
- Debug Readback 只用于抽样验证。

## Implementation

- Shader 第一版使用单 pass atomic append；CPU builder 同步生成 visible list oracle。
- 每 Draw Group 维护 visible count/offset，CPU oracle 同步生成 indirect inputs。
- objectCount/capacity 在 CPU builder 中有上限标志。
- 真实 GPU counter reset 仍等待 RenderGraph 多写/clear 语义。

## Unit Tests

- 小固定场景 CPU/GPU visible count 一致。
- 全可见、全剔除、边界 AABB、空场景。
- 超容量设置错误标志且不写越界。
- Validation Layer 无 storage/barrier 错误。

## Boundaries

- 不提交 Indirect Draw。
- 不做 Occlusion。
- 不承诺 atomic append 是最终算法。
- 不声明 GPU culling 已替代 CPU culling；当前仍是 CPU oracle + shader contract。

## Knowledge and Design Notes

- 先做 CPU oracle 不是倒退，而是 GPU-driven 正确性的前提。没有 oracle 时，GPU cull 错误通常只表现为“物体消失”，无法定位是 bounds、frustum、barrier、counter 还是 indirect argument 错。
- visible list 使用 object id/hash 进行比较，不依赖 vector 地址或 CPU 排序副作用。
- 当前空场景 smoke 也有价值：它验证路径选择、资产编译、graph/diagnostics 和 schema，而不是性能。
