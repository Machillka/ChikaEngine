## Implementation Record

- Runner 实现 `Warmup -> Sampling -> Flushing -> Complete`，严格保存指定数量的 sample frames。
- Statistics 使用 nearest-rank 计算 min/max/mean/p50/p95/p99，拒绝 NaN、Infinity 和负时间。
- Vulkan timestamp scope 记录提交 `frameIndex`；Runner 用该 ID 回填原 CPU sample，flush 帧负责接收采样尾部 GPU 结果。
- 无 GPU 结果的样本不写入 0 伪装有效数据，而是增加 `missingGpuSamples`。

## Changed Files and Reasons

- `BenchmarkRunner.hpp/.cpp`：状态机、Frame ID 索引和 delayed GPU 回填。
- `SampleStatistics.hpp/.cpp`：与 Engine 无关的纯统计函数。
- `RenderDiagnostics.hpp`、`VulkanRHIDevice.hpp/.cpp`：把异步 query 结果关联到原始提交帧。
- `tests/unit/BenchmarkTests.cpp`：用已知两帧延迟序列验证 warmup 排除、tail flush 和 GPU 不错配。

## Knowledge and Design Notes

- **Warmup**：排除首次资产上传、Pipeline 创建、缓存填充和频率爬升；它不是删除不喜欢的异常帧。
- **p50/p95/p99**：平均值会掩盖长尾；p50 描述典型成本，p95/p99 描述用户更容易感知的卡顿风险。
- **GPU 是异步时间线**：CPU 当前帧读取到的 query 通常属于更早提交，必须用 Frame ID join，不能按“读取时刻”归属。
- **Flush 的意义**：停止新增 CPU sample 后继续推进若干帧，只为收齐最后的 GPU query；flush 帧本身不进入分布。
